# 优化建议功能实现方案

## 需求目标

为 AgentSprint 增加面向全体项目参与人的优化建议闭环：

- 任意项目参与人可以选择项目、端、模块和可选需求提交优化建议。
- 需求管理提供“需求转化”能力，可以从现有需求反馈和优化建议中选择内容，生成或补充需求，并将备注追加到需求描述中，格式为新起一行 `追加备注: xxxx`。
- 需求管理列表的行展开从当前“反馈与子需求混合列表”调整为两行 table：第一行展示需求拆解任务，第二行展示反馈与子需求，支持针对已完成任务提前提交反馈。

## 现状确认

后端已有基础能力：

- 数据模型已包含 `SprintFeatureSuggestionEntity`，表名为 `sprint_feature_suggestion`，字段包含 `ProjectId`、`EndpointId`、`ModuleId`、`RequirementId`、`Content`、`Status`、`CreatedBy`。
- DTO 已包含 `CreateSprintFeatureSuggestionRequest` 和 `SprintFeatureSuggestionResult`。
- 服务已包含 `CreateFeatureSuggestionAsync` 和 `ListFeatureSuggestionsAsync`，会校验项目、端、模块、需求归属，并要求提交人是项目参与人。
- 控制器已暴露 `POST /mvp/suggestions` 和 `GET /mvp/suggestions`。
- 前端 API 已包含 `CreateFeatureSuggestionRequest`、`FeatureSuggestion`、`createFeatureSuggestionApi`、`listFeatureSuggestionsApi`。

主要缺口：

- 没有独立优化建议页面、菜单入口和表单流程。
- 优化建议只有创建和查询，没有“转化为需求”或“被采纳后关联需求”的动作接口。
- 现有反馈转需求仅支持单条需求反馈转新需求，不能选择多条反馈与优化建议，也不能追加备注。
- 需求展开区当前为反馈与子需求混合列表，没有展示拆解任务，也不能基于已完成任务提交反馈。
- 需求反馈模型只绑定 `RequirementId`，不支持绑定到具体 `DevelopmentTaskId`。

## 后端方案

### 数据模型

建议保留现有 `SprintFeatureSuggestionEntity`，增加必要字段：

- `Title`：建议标题，便于列表展示和转化选择。
- `ConvertedRequirementId`：建议被转化后的需求 ID。
- `ConvertedAt`：转化时间。
- 可选 `ClosedAt`：关闭时间。

建议扩展 `SprintRequirementFeedbackEntity`：

- `DevelopmentTaskId`：可空，表示反馈针对整个需求或某个拆解任务。

如需要更清晰的转化审计，可增加 `sprint_requirement_conversion_source` 表，记录一次转化选择的多条来源：

- `RequirementId`
- `SourceType`：`feedback` 或 `suggestion`
- `SourceId`
- `CreatedBy`

若第一阶段要控制范围，也可以先通过 `SprintRequirement.SourceFeedbackId` 和 `SprintFeatureSuggestion.ConvertedRequirementId` 表达单源转化，多源选择时将来源内容合并进描述。

### DTO

新增或调整：

- `CreateSprintFeatureSuggestionRequest` 增加 `Title`。
- `CreateSprintRequirementFeedbackRequest` 增加可选 `DevelopmentTaskId`。
- 新增 `ConvertRequirementSourcesRequest`：
  - `Title`
  - `Description`
  - `Priority`
  - `Stakeholders`
  - `FeedbackIds`
  - `SuggestionIds`
  - `Remark`
  - `EndpointId`
  - `ModuleId`

### 接口

建议保留已有接口：

- `POST /mvp/suggestions`
- `GET /mvp/suggestions?projectId=&moduleId=&requirementId=`

新增接口：

- `POST /mvp/requirements/{id}/feedback`
  - 请求体扩展 `developmentTaskId`，服务端校验任务属于该需求。
- `POST /mvp/requirements/{id}/convert-sources`
  - 从指定需求上下文选择反馈和优化建议转化为新需求或补充需求。
  - 服务端校验反馈属于该需求，建议属于同项目且未转化。
  - 备注追加规则：在最终需求描述末尾追加空行和 `追加备注: {Remark}`。
  - 转化成功后将反馈置为 `converted`，将建议置为 `accepted` 并写入 `ConvertedRequirementId`、`ConvertedAt`。

可选补充：

- `POST /mvp/suggestions/{id}/close`
- `POST /mvp/suggestions/{id}/accept`

### 服务规则

- 所有人都可提交建议应理解为“所有项目参与人”，沿用当前 `EnsureProjectParticipantAsync`。如果确实要开放到任意登录用户，需要明确是否允许非项目成员看到项目、端、模块、需求下拉数据。
- 建议与需求绑定时，必须校验 `Requirement.ProjectId == Suggestion.ProjectId`，且端、模块归属一致或可为空。
- 针对任务提交反馈时，任务必须属于当前需求；已完成任务优先开放，是否允许进行中任务反馈需产品确认。
- 多源转化时应避免重复转化同一条 open feedback 或 accepted suggestion。

## 前端方案

### 新增优化建议页面

路径建议：

- 路由：`/sprint/suggestions`
- 页面：`src/admin/apps/web-tdesign/src/views/sprint/suggestions/index.vue`
- 菜单分组：产品管理，标题“优化建议”，图标可用 `lucide:lightbulb`

页面结构：

- 顶部筛选：项目、端、模块、需求、状态。
- 列表：标题、项目、端、模块、关联需求、状态、提交人、提交时间。
- 新增建议抽屉：项目、端、模块、需求、标题、内容。
- 详情抽屉：展示建议内容和关联信息。

### 需求管理转化能力

在需求管理页增加“需求转化”入口：

- 从需求详情或展开区进入。
- 弹窗或抽屉中加载：
  - 当前需求的 open 反馈。
  - 当前项目下 open 优化建议，可按当前需求、模块、端默认筛选。
- 支持多选反馈和建议。
- 表单字段：标题、优先级、干系人、需求内容、追加备注。
- 提交时调用 `POST /mvp/requirements/{id}/convert-sources`。

描述拼接建议：

- 初始描述可由所选反馈和建议内容合并生成。
- 用户填写备注后，提交给后端统一追加，避免前端和后端重复拼接。

### 需求列表展开区调整

当前 `requirements/index.vue` 的展开区使用 `getRequirementFollowUpItems` 混合展示反馈和子需求。建议改为两个 table：

- 第一行 table：拆解任务。
  - 数据来源：`listDevelopmentTasksApi({ requirementId })`。
  - 列：任务标题、状态、负责人、优先级、完成时间、操作。
  - 操作：已完成任务显示“提交反馈”，打开反馈抽屉并带入 `developmentTaskId`。
- 第二行 table：反馈与子需求。
  - 数据来源：现有 `listRequirementFeedbackApi` 和 `childRequirementsBySource`。
  - 反馈列展示关联任务标题；子需求保持原有详情、提交评审、拆解、关闭等操作。

前端状态需要新增：

- `requirementTaskMap: Record<string, DevelopmentTask[]>`
- 展开行时并行加载任务和反馈。
- 反馈表单增加隐藏或只读的 `developmentTaskId` 上下文。

## 测试范围

后端单元测试：

- 创建建议成功：项目参与人、项目/端/模块/需求归属校验。
- 非项目参与人提交建议失败。
- 多源转化成功：生成需求、追加备注、反馈和建议状态更新。
- 重复转化 open/accepted 状态保护。
- 任务反馈：任务属于需求时成功，不属于需求时失败。

控制器测试：

- `POST /mvp/suggestions` 使用认证用户。
- `GET /mvp/suggestions` 透传筛选条件。
- `POST /mvp/requirements/{id}/convert-sources` 使用路由需求和认证用户。

前端检查：

- `corepack pnpm -F @vben/web-tdesign run typecheck`
- 新页面和需求管理页的类型覆盖建议、任务反馈和转化表单。

## 当前实现状态

本次实现已完成优化建议闭环中的后端服务、需求管理页面联动和验证覆盖：

- `SprintRequirementFeedbackEntity` 增加 `DevelopmentTaskId`，支持对已完成拆解任务提前提交回馈。
- `SprintFeatureSuggestionEntity` 增加 `ConvertedRequirementId`、`ConvertedAt`，建议被转化后置为 `accepted` 并记录目标需求。
- 新增 `ConvertRequirementSourcesRequest` / `ConvertRequirementSourcesAsync` / `POST /mvp/requirements/{id}/convert-sources`，支持选择多条 open 回馈和优化建议生成后续草稿需求。
- 旧的 `POST /mvp/requirements/{id}/feedback/{feedbackId}/convert` 保持兼容，内部复用统一来源转化逻辑。
- 服务端统一追加备注，规则为在描述末尾追加空行和 `追加备注: xxxx`。
- 需求管理展开区展示拆解任务和回馈/子需求两个区块，已完成任务可直接打开回馈抽屉并带入 `developmentTaskId`。
- 需求转化抽屉支持多选回馈来源、多选建议来源、填写追加备注，并同步刷新回馈与建议状态。
