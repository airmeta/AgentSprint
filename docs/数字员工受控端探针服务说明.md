# 数字员工受控端探针服务说明

## 定位

`AgentSprint.Worker` 是数字员工受控端的第一阶段控制台宿主，基于 `Air.Cloud.HostApp` 启动，不提供 HTTP API。它用于承载一个 Codex 数字员工实例，并在后续阶段接入 AgentSprint 主平台的注册、心跳、命令和任务领取闭环。

## 启动入口

项目路径：

```text
src/api/AgentSprint.Worker/
```

入口使用 HostApp 注入：

```csharp
var hostBuilder = Host.CreateDefaultBuilder(args);
hostBuilder.HostInjectInFile(Assembly.GetExecutingAssembly());
```

该入口只使用 `Air.Cloud.HostApp`，不混用 `Air.Cloud.WebApp`，避免同一入口项目出现多个 `IAppInjectStandard` 实现。

## 当前能力

- 读取 `Worker` 和 `AgentSprint` 配置段。
- 检查并创建 `WorkspaceRoot` 和 `RunsRoot`。
- 检查 `CodexHome/config.toml` 是否存在。
- 执行环境探针：
  - `codex --version`
  - `git --version`
  - `dotnet --version`
  - `node --version`
  - `codex login status`
- 可通过 `Worker:RunSmokeOnStartup=true` 启动一次 `codex exec` smoke run。
- 每次 run 落盘：
  - `prompt.txt`
  - `stdout.log`
  - `stderr.log`
  - `final.md`
  - `run.json`

## 配置示例

```json
{
  "Worker": {
    "WorkerId": "worker-1",
    "WorkerName": "agentsprint-codex-worker-1",
    "WorkspaceRoot": "/workspaces",
    "RunsRoot": "/runs",
    "CodexHome": "/codex-home",
    "PollIntervalSeconds": 15,
    "MaxRunMinutes": 60,
    "SandboxMode": "workspace-write",
    "RunSmokeOnStartup": false,
    "SmokePrompt": "你好"
  },
  "AgentSprint": {
    "ApiBaseUrl": "http://api:5000",
    "AgentToken": ""
  }
}
```

## 验证方式

待本机 .NET 10 运行时安装完整后，在 `src/api` 目录执行：

```powershell
dotnet test .\AgentSprint.Tests\AgentSprint.Tests.csproj
```

开启 smoke run 时可设置：

```powershell
$env:Worker__RunSmokeOnStartup = "true"
dotnet run --project .\AgentSprint.Worker\AgentSprint.Worker.csproj
```

## Standalone Deployment Contract

`AgentSprint.Worker` should be deployed as a generic controlled endpoint. The container only needs two platform identity settings:

```yaml
environment:
  AgentSprint__ApiBaseUrl: "http://api:5000"
  AgentSprint__AgentToken: "${AGENTSPRINT_AGENT_TOKEN}"
```

`WorkerId`, `WorkerName`, workspace paths, run-log paths, `CodexHome`, MCP endpoint, polling cadence, max run minutes, sandbox mode, smoke policy, Codex provider/model/base URL, and similar runtime settings come from the `digital_worker` management record. On startup the Worker calls `GET /worker-runtime/config` with the Agent Token in `Authorization: Bearer ...`; the platform validates `sys_agent_token`, finds the bound `digital_worker.AgentTokenId`, and returns the full runtime config.

Each digital worker still needs its own container, Agent Token, and mounted directories. Mounts are container resources, so compose or the container platform still declares them, while worker identity and runtime behavior stay in digital-worker management.

## Prompt Runtime Context

The platform prepends a non-secret digital-worker runtime context to every work prompt returned by `GET /worker-runtime/work/{targetType}/{targetId}/prompt`. The context includes worker identity, workspace root, resolved project workspace path, runs/log root, `CodexHome`, sandbox mode, Codex provider/model, polling limits, concurrency limits, config version, bound project IDs, endpoint IDs, skill IDs, and worker description.

Before `AgentSprint.Worker` calls `codex exec`, it also prepends the current run context to the actual prompt written to `prompt.txt`. That section includes the run ID, working directory, run directory, `prompt.txt`, `stdout.log`, `stderr.log`, `final.md`, `run.json`, Codex Home, sandbox mode, model, timeout settings, and prepared Git branch/commit state.

Sensitive values are intentionally excluded from both prompt layers. Agent Token values, Git access tokens, API keys, database passwords, and SSH private keys must stay in Worker process memory or Codex auth/config files and must not be rendered into the prompt.

## Worker Git Workspace Preparation

Before a `start_task` or `start_bug` command enters `codex exec`, `AgentSprint.Worker` prepares the project workspace with `GitWorkspaceManager`.

Flow:

1. Worker reads `projectCode`, `repositoryUrl` and optional `branch` from the command payload or from `GET /worker-runtime/work/{targetType}/{targetId}/prompt`.
2. Worker resolves the local path as `{WorkspaceRoot}/{projectCode}`.
3. If the directory is missing or empty, Worker runs `git clone`.
4. If the directory already contains `.git`, Worker runs `git remote set-url origin` when needed, then `git fetch --prune origin`, branch checkout, and `git pull --ff-only`.
5. Worker reports `workspace_prepared` with `workspacePath`, `repositoryAvailable`, `branch`, `commit` and `dirty`.
6. Only after the workspace is ready does Worker start `codex exec`; real task and bug runs no longer use `--skip-git-repo-check`.

If the project has no repository URL, Worker creates the workspace directory but marks the run as `blocked` before starting Codex. This keeps real development tasks from running in an empty folder.

Security boundary:

- `repositoryReference` remains the sanitized value used in prompt text.
- `repositoryUrl` is returned in the prompt context for Worker only and is used directly by the Git tool.
- Worker does not write the real repository URL into `prompt.txt` or ask Codex to run the initial clone/fetch/pull step.
