# 权限管理模型后续演进方案

## 目的

本文记录参考项目 `air.micro.temp/Services/Security` 的权限模型设计，并说明 AgentSprint 当前阶段的落地边界。

最小 MVP 登录、菜单、权限码闭环已经跑通。当前阶段不再停留在最小 RBAC，而是推进到“RBAC + 通用关联表 + 基础系统管理”能力，便于维护人员管理用户、角色、菜单下按钮权限、部门和岗位。

## 当前已实施范围

当前 AgentSprint 已保留原最小 RBAC 表，并增加通用关联表承载扩展授权关系。

已实施主数据：

- 用户：`sys_user`
- 角色：`sys_role`
- 菜单：`sys_menu`
- 权限码：`sys_permission`
- 用户组：`sys_user_group`
- 角色组：`sys_role_group`
- 部门：`sys_department`
- 岗位：`assignment`
- 通用实体关联：`sys_entity_association`

仍保留兼容表：

- 用户角色：`sys_user_role`
- 角色菜单：`sys_role_menu`
- 角色权限：`sys_role_permission`

兼容策略：

- 新系统管理写入会同步 `sys_entity_association` 和旧 RBAC 关系表。
- 登录、菜单、权限码解析优先读取通用关联表。
- 当通用关联表没有对应关系时，继续回退读取旧 RBAC 关系表，保证 MVP 流程不被破坏。
- 数据库初始化会把旧 `UserRole`、`RoleMenu`、`RolePermission` 回填为通用关联关系。
- 权限码通过 `sys_permission.MenuId` 归属到菜单，管理端不再开放独立权限码菜单，统一在“菜单管理 -> 按钮权限”中维护。

## 通用关联表

当前 `sys_entity_association` 承载本阶段需要的授权和组织关系。

字段：

- `SourceEntityId`：源实体 ID
- `TargetEntityId`：目标实体 ID
- `AssociationType`：关联类型

当前支持的关联类型：

- `user_user_group`：用户与用户组
- `user_role_group`：用户与角色组
- `user_role`：用户与角色
- `user_department`：用户与部门
- `user_assignment`：用户与岗位
- `user_group_role_group`：用户组与角色组
- `user_group_role`：用户组与角色
- `role_role_group`：角色与角色组
- `role_menu`：角色与菜单
- `role_permission`：角色与权限码

授权解析链路：

```text
User
  -> direct Role
  -> UserGroup -> Role
  -> UserGroup -> RoleGroup -> Role
  -> RoleGroup -> Role
  -> Role -> Menu / Permission
```

## 系统管理接口

当前新增系统管理接口统一挂载在 `/system`，需要登录后访问。

- `GET /system/users`
- `POST /system/users`
- `DELETE /system/users/{id}`
- `GET /system/roles`
- `POST /system/roles`
- `DELETE /system/roles/{id}`
- `GET /system/menus`
- `POST /system/menus`
- `DELETE /system/menus/{id}`
- `GET /system/permissions`：兼容接口，供菜单管理页和角色授权读取按钮权限
- `POST /system/permissions`：保存菜单下按钮权限，`MenuId` 必填且必须指向有效菜单
- `DELETE /system/permissions/{id}`：删除按钮权限
- `GET /system/user-groups`
- `POST /system/user-groups`
- `DELETE /system/user-groups/{id}`
- `GET /system/role-groups`
- `POST /system/role-groups`
- `DELETE /system/role-groups/{id}`
- `GET /system/departments`
- `POST /system/departments`
- `DELETE /system/departments/{id}`
- `GET /system/assignments`
- `POST /system/assignments`
- `DELETE /system/assignments/{id}`
- `GET /system/associations`
- `POST /system/associations`
- `DELETE /system/associations/{id}`

管理端页面：

- `/system/users`：用户维护和用户角色授权
- `/system/roles`：角色维护、角色菜单授权、角色按钮权限授权
- `/system/menus`：菜单维护，按父子级展示树型表格，便于维护目录、菜单、隐藏路由和菜单下按钮权限
- `/system/departments`：部门维护
- `/system/assignments`：岗位维护

用户、角色、菜单、按钮权限、部门、岗位基础维护页面已提供“查询 / 重置”和本地检索条件，用于按关键词、状态、类型、角色或授权状态快速定位数据。

删除行为均为软删除。删除菜单时会同步软删除直接挂载在该菜单下的按钮权限。用户组、角色组和通用关联接口作为底层模型能力暂时保留，但当前管理菜单不开放这三项维护入口。

## 当前明确不实施

以下能力属于大规模分布式微服务架构阶段，本阶段不加入：

- 应用管理：`SYS_APP`
- 动作权限：`SYS_ACTION_PERMISSION`
- 应用路由：`APP_ROUTE`
- 路由授权：`APP_ROUTE_AUTH`
- 基于应用维度的 `APP_ID` 隔离
- 网关或接口路由级授权校验
- 权限缓存与跨服务变更刷新

当前项目聚焦 RBAC、组织辅助授权和通用关联表，不引入应用、动作权限、应用路由和路由授权。

## 后续演进建议

当当前系统管理能力稳定后，再按业务规模评估下一阶段：

1. 引入权限缓存和授权关系变更刷新机制。
2. 在多应用或微服务拆分明确后，再引入 `SYS_APP` 和应用维度隔离。
3. 在需要网关或接口级统一授权时，再引入动作权限、应用路由和路由授权。
4. 补充审计日志，记录授权关系创建、删除、恢复和主数据变更。

## 验证方式

后端验证：

```powershell
dotnet test src\api\AgentSprint.slnx --no-restore
```

前端验证：

```powershell
corepack pnpm -F @vben/web-tdesign run typecheck
```

本地联调：

1. 使用开发账号 `admin / 123456` 登录。
2. 访问 `/system/users`、`/system/roles`、`/system/menus`、`/system/departments`、`/system/assignments`。
3. 验证列表加载、新增、编辑、删除、查询、重置和分页。
4. 在菜单管理页确认系统菜单按树型表格展示，点击“按钮权限”可维护当前菜单下的权限码。
5. 在角色管理页确认按钮权限选项带有所属菜单，并可随角色保存。
