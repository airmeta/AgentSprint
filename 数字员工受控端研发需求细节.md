# 数字员工受控端研发需求细节

## 1. 总体说明

数字员工受控端是运行在独立容器中的 .NET 10 Worker 程序，用来驱动本容器内的 Codex CLI，并与 AgentSprint 主平台形成闭环。

每个数字员工容器只处理一个 Worker 实例：

```text
1 container = 1 digital worker = 1 CODEX_HOME = 1 workspace root = 1 runs root = 1 Agent identity
```

主平台负责：

- 数字员工身份。
- 会话。
- 心跳。
- 命令。
- 任务租约。
- 任务状态。
- 缺陷状态。
- 运行记录。
- 审计事件。

受控端负责：

- 启动后注册自己。
- 定时心跳。
- 拉取平台命令或领取任务。
- 准备工作区。
- 调用 `codex exec`。
- 采集运行日志和最终响应。
- 按结果回写主平台。

第一版通信方式：

```text
HTTP register + HTTP heartbeat + HTTP command polling + HTTP event/report callback
```

后续可增加 WebSocket/SignalR 唤醒，但数据库中的 Session、Command、Run、Lease、Event 仍是事实源。

## 2. 受控端基础配置需求

### REQ-WORKER-001：读取数字员工运行配置

目标：

Worker 启动时读取本容器所属数字员工的基础配置，形成运行上下文。

数据来源：

- 环境变量。
- `appsettings.json`。
- Docker Compose `.env`。
- `/codex-home/config.toml`。

建议环境变量：

| 变量 | 说明 |
| --- | --- |
| `WORKER_ID` | 数字员工唯一编码。 |
| `WORKER_NAME` | 数字员工名称。 |
| `AGENTSPRINT_API_BASE_URL` | 主平台 API 地址。 |
| `AGENTSPRINT_AGENT_TOKEN` | 主平台 Agent Token。 |
| `CODEX_HOME` | Codex 配置目录，默认 `/codex-home`。 |
| `WORKSPACE_ROOT` | 工作区根目录，默认 `/workspaces`。 |
| `RUNS_ROOT` | 运行记录目录，默认 `/runs`。 |
| `WORKER_PROJECT_ID` | 限定项目，可为空。 |
| `WORKER_PROJECT_CODE` | 限定项目编码，可为空。 |
| `WORKER_POLL_INTERVAL_SECONDS` | 空闲轮询间隔。 |
| `WORKER_MAX_RUN_MINUTES` | 单次 Codex 执行最大时长。 |

获取方式：

- Worker 使用 .NET `IConfiguration` 读取环境变量和配置文件。
- 所有路径转为绝对路径。
- 启动时校验必填字段。

处理逻辑：

- 如果 `AGENTSPRINT_API_BASE_URL` 为空，启动失败。
- 如果 `AGENTSPRINT_AGENT_TOKEN` 为空，允许进入本地测试模式，但不能领取平台任务。
- 如果 `CODEX_HOME/config.toml` 不存在，启动失败或进入 degraded 状态。
- 如果 `/workspaces`、`/runs` 不存在，自动创建。

回给主平台：

- 注册时携带 `worker_id`、`worker_name`、`workspace_root`、`codex_home`、`runs_root`、`project_id`、`project_code`。

验收标准：

- 修改 `.env` 后重启容器，Worker 能读取新配置。
- 缺少关键配置时日志明确说明缺失项。
- 不把 API Key 或 Agent Token 打到日志中。

### REQ-WORKER-002：校验 Codex 运行环境

目标：

Worker 启动时确认容器具备执行 Codex 任务的基础能力。

数据来源：

- 容器内命令行。
- Codex 配置目录。
- Codex 登录状态。

获取方式：

Worker 启动时执行：

```bash
codex --version
git --version
dotnet --version
node --version
codex login status
```

处理逻辑：

- 收集每个命令的退出码、stdout、stderr。
- `codex --version` 失败则 Worker 不进入工作循环。
- `codex login status` 未登录时，标记为 `auth_required`。
- 如果环境变量存在 `OPENAI_API_KEY`，可执行一次非交互登录：

```bash
printenv OPENAI_API_KEY | codex login --with-api-key
```

回给主平台：

- 上报 `worker_environment_checked` 事件。
- 上报字段：`codex_version`、`git_version`、`dotnet_version`、`node_version`、`login_status`、`error`。

验收标准：

- 环境正常时 Worker 状态为 `idle`。
- 未登录时 Worker 状态为 `auth_required`，不领取任务。
- 日志中不出现完整 API Key。

## 3. 会话与心跳需求

### REQ-WORKER-003：注册数字员工会话

目标：

Worker 启动后向主平台注册一个可追踪的会话。

数据来源：

- Worker 配置。
- 主平台用户和 Agent Token。
- 容器运行信息。

获取方式：

Worker 调用主平台 API：

```text
POST /worker/sessions/register
```

建议请求：

```json
{
  "workerId": "worker-1",
  "workerName": "math-codex-worker-1",
  "containerName": "math-codex-worker-1",
  "hostName": "container-hostname",
  "projectId": "",
  "projectCode": "",
  "workspaceRoot": "/workspaces",
  "runsRoot": "/runs",
  "codexHome": "/codex-home",
  "codexVersion": "codex-cli 0.139.0",
  "sandboxMode": "workspace-write"
}
```

处理逻辑：

- 主平台校验 Agent Token。
- 主平台创建或更新 `DigitalWorker`。
- 主平台创建 `WorkerSession`。
- 如果同一个 `workerId` 有未关闭 session，标记旧 session 为 `expired` 或 `replaced`。

回给主平台：

- Worker 发起注册请求。
- 主平台返回：

```json
{
  "sessionId": "session-xxx",
  "status": "online",
  "heartbeatIntervalSeconds": 15,
  "maxRunMinutes": 60,
  "offlineRequested": false,
  "commands": []
}
```

验收标准：

- 容器启动后主平台可看到该数字员工在线。
- 重启容器后生成新 session，旧 session 不再处于 active。

### REQ-WORKER-004：发送心跳并接收命令

目标：

Worker 定时上报当前状态，并从主平台获取待执行命令。

数据来源：

- Worker 当前内存状态。
- 当前 Run 状态。
- 当前任务或缺陷。
- 主平台命令队列。

获取方式：

Worker 定时调用：

```text
POST /worker/sessions/{sessionId}/heartbeat
```

建议请求：

```json
{
  "workerId": "worker-1",
  "status": "idle",
  "currentRunId": null,
  "currentTaskType": null,
  "currentTaskId": null,
  "idleRound": 0,
  "lastError": null
}
```

处理逻辑：

- Worker 根据当前状态填充 `online`、`idle`、`busy`、`auth_required`、`error`、`offline`。
- 主平台更新 `lastHeartbeatAt`。
- 主平台返回待处理命令。
- Worker 对命令做幂等处理，已 ACK 的命令不重复执行。

主平台返回：

```json
{
  "accepted": true,
  "offlineRequested": false,
  "stopCurrentRun": false,
  "polling": {
    "shouldContinue": true,
    "nextIntervalSeconds": 30,
    "idleRound": 1
  },
  "commands": [
    {
      "id": "cmd-001",
      "type": "claim_next_work",
      "payload": {
        "projectId": "project-1"
      }
    }
  ]
}
```

回给主平台：

- 每次心跳更新 Worker 状态。
- 收到命令后调用命令 ACK 接口：

```text
POST /worker/commands/{commandId}/ack
```

验收标准：

- Worker 心跳间隔符合平台返回的 `nextIntervalSeconds`。
- 平台请求 `offlineRequested=true` 时，Worker 完成当前任务后下线。
- 平台请求 `stopCurrentRun=true` 时，Worker 尝试取消当前 Codex 进程。

## 4. 命令与任务领取需求

### REQ-WORKER-005：拉取命令并分类处理

目标：

Worker 能处理主平台下发的标准命令。

数据来源：

- 心跳返回的 `commands`。
- 主平台命令查询接口。

命令类型：

| 命令 | 说明 |
| --- | --- |
| `claim_next_work` | 领取下一项工作。 |
| `start_task` | 启动指定开发任务。 |
| `start_bug_fix` | 启动指定缺陷修复。 |
| `cancel_current_run` | 取消当前 Codex 执行。 |
| `stop_after_current` | 当前任务完成后下线。 |
| `reload_config` | 重新读取平台配置。 |
| `go_offline` | 协作式下线。 |
| `clean_workspace` | 清理指定工作区。 |

处理逻辑：

- Worker 对命令做幂等检查。
- `busy` 状态下只接受取消、停止、心跳类命令。
- `idle` 状态下可以领取新任务。
- 每个命令必须写入处理结果。

回给主平台：

```text
POST /worker/commands/{commandId}/complete
POST /worker/commands/{commandId}/fail
```

验收标准：

- 同一命令重复投递不会重复执行 Codex。
- 不支持的命令被标记为 `unsupported`，并写入原因。

### REQ-WORKER-006：领取下一项工作

目标：

Worker 空闲时按平台规则领取下一项任务或缺陷。

数据来源：

- Worker 配置中的项目范围。
- 主平台任务大厅。
- 主平台“我的任务”。
- 主平台缺陷列表。
- 任务租约。

获取方式：

第一版可通过 MCP 同等逻辑对应的主平台 API 实现，或复用现有 MCP 工具语义：

```text
claim_next_work(project_id, requirement_id, owner_device, primary_only, session_id, idle_round)
```

如果 Worker 直接调用 AgentSprint API，建议新增：

```text
POST /worker/sessions/{sessionId}/claim-next-work
```

处理逻辑：

平台优先级：

1. 当前用户已完成需求下仍未关闭的缺陷。
2. 当前用户未完成任务，按 `in_progress`、`assigned`、`pending_assign`。
3. 当前用户可见的待分配任务大厅任务。

Worker 处理：

- 如果返回 `kind=none`，按 `polling.nextIntervalSeconds` 等待。
- 如果返回 `kind=task`，创建 development task run。
- 如果返回 `kind=bug`，创建 bug fix run。
- 如果返回 `claim`，保存租约信息到 run。

回给主平台：

- 领取成功时平台创建或更新 `TaskLease`。
- Worker 创建 `WorkerRun`。
- Worker 上报 `task_claimed` 或 `bug_claimed` 事件。

验收标准：

- 两个 Worker 同时领取时，不会拿到同一任务。
- 领取成功后平台能看到 active lease。
- 无任务时 Worker 不报错，按策略轮询。

## 5. 工作区准备需求

### REQ-WORKER-007：准备项目工作区

目标：

Worker 在执行 Codex 前准备项目代码目录。

数据来源：

- 任务返回的 `projectId`、`projectCode`、`repositoryReference`。
- 主平台项目配置。
- Worker 本地 `/workspaces`。

获取方式：

Worker 从领取结果或任务详情读取：

| 字段 | 用途 |
| --- | --- |
| `projectId` | 平台项目标识。 |
| `projectCode` | 本地目录名。 |
| `repositoryReference` | Git 仓库地址或脱敏引用。 |
| `branch` | 工作分支，可为空。 |

处理逻辑：

- 工作区路径为 `/workspaces/{projectCode}`。
- 如果目录不存在且有仓库地址，则执行 `git clone`。
- 如果目录存在，则执行 `git fetch`。
- 如果无仓库地址，则创建空目录，仅允许测试 prompt。
- 根据任务创建或切换分支。
- 检查当前工作区是否有未提交改动。

回给主平台：

- 上报 `workspace_prepared` 事件。
- 事件 payload 包含 `workspacePath`、`repositoryAvailable`、`branch`、`commit`、`dirty`。

验收标准：

- 有仓库地址时能准备 Git 工作区。
- 无仓库地址时能进入测试模式，但不能执行真实开发任务。
- 工作区异常时 run 标记为 `workspace_failed`。

## 6. Codex 执行需求

### REQ-WORKER-008：生成 Codex 执行提示词

目标：

Worker 根据领取到的工作项生成最小启动提示词。

数据来源：

- `claim_next_work` 返回的 `item`。
- 主平台任务 ID 或缺陷 ID。
- Worker session ID。
- MCP endpoint。

处理逻辑：

开发任务 prompt：

```text
你是 AgentSprint 数字员工。
请先调用 agentsprint.register_session 注册当前会话。
然后调用 agentsprint.get_task_prompt，参数 task_id = "<taskId>"。
以 get_task_prompt 返回的 task_detail、requirement_detail、skill_pack 为准完成任务。
完成代码修改和验证后，调用 agentsprint.complete_my_task，参数 task_id = "<taskId>"。
如果遇到无法继续的问题，说明阻塞原因并停止，不要伪造完成状态。
```

缺陷修复 prompt：

```text
你是 AgentSprint 数字员工。
请先调用 agentsprint.register_session 注册当前会话。
缺陷 ID 为 "<bugId>"。
请根据平台缺陷信息完成修复和本地验证。
完成后调用 agentsprint.fix_bug，参数 bug_id = "<bugId>"。
如果缺陷信息不足，说明阻塞原因并停止。
```

回给主平台：

- 创建 run 时保存 prompt 摘要，不保存敏感令牌。
- 上报 `codex_prompt_created` 事件。

验收标准：

- Prompt 中不出现 API Key、Agent Token、数据库密码、SSH 私钥。
- Prompt 只包含任务标识和执行顺序，不内嵌长需求正文。

### REQ-WORKER-009：启动 codex exec

目标：

Worker 启动 Codex CLI 执行任务，并持续监控进程。

数据来源：

- Worker 配置。
- Run 对象。
- Prompt。
- 工作区路径。

执行命令：

```bash
codex exec \
  --cd /workspaces/<project-code> \
  --sandbox workspace-write \
  --output-last-message /runs/<run-id>/final.md \
  "<prompt>"
```

测试模式可追加：

```bash
--skip-git-repo-check
```

处理逻辑：

- 创建 `/runs/{runId}`。
- 将 prompt 写入 `/runs/{runId}/prompt.txt`。
- stdout 写入 `/runs/{runId}/stdout.log`。
- stderr 写入 `/runs/{runId}/stderr.log`。
- final response 写入 `/runs/{runId}/final.md`。
- 进程超时则终止。
- 退出码非 0 时标记失败。

回给主平台：

- 启动前上报 `codex_started`。
- 执行中分段上报 `codex_output_chunk`。
- 结束后上报 `codex_finished`。

验收标准：

- 成功执行 `你好` 并获得响应。
- stdout、stderr、final.md 均落盘。
- 超时能终止进程并回写状态。

### REQ-WORKER-010：解析 Codex 执行结果

目标：

Worker 将 Codex 进程结果转换为平台可理解的 run 结果。

数据来源：

- Codex 退出码。
- `/runs/{runId}/stdout.log`。
- `/runs/{runId}/stderr.log`。
- `/runs/{runId}/final.md`。
- Codex session jsonl。

处理逻辑：

结果分类：

| 分类 | 条件 |
| --- | --- |
| `success` | 退出码为 0，final.md 存在，未检测到阻塞关键词。 |
| `codex_failed` | Codex 退出码非 0。 |
| `timeout` | 超过 `MaxRunMinutes`。 |
| `cancelled` | 平台取消命令触发。 |
| `blocked` | final.md 明确说明缺少信息、权限不足或需人工判断。 |
| `mcp_failed` | MCP 必需且连接失败。 |
| `verification_failed` | 验证命令失败。 |
| `lease_lost` | 平台租约丢失或过期。 |

回给主平台：

```json
{
  "runId": "run-xxx",
  "status": "success",
  "exitCode": 0,
  "finalMessage": "完成摘要",
  "stdoutPath": "/runs/run-xxx/stdout.log",
  "stderrPath": "/runs/run-xxx/stderr.log",
  "finalPath": "/runs/run-xxx/final.md",
  "startedAt": "...",
  "completedAt": "..."
}
```

验收标准：

- 平台能展示 run 状态、耗时、最终消息和日志路径。
- 失败时有明确分类和错误摘要。

## 7. 回写主平台需求

### REQ-WORKER-011：开发任务完成回写

目标：

开发任务完成并验证通过后，将任务状态回写给主平台。

数据来源：

- Run 结果。
- Codex final.md。
- 任务 ID。
- 租约信息。

获取方式：

优先由 Codex 通过 MCP 调用：

```text
agentsprint.complete_my_task(task_id)
```

Worker 兜底可直接调用主平台 API：

```text
POST /mvp/tasks/{id}/complete
```

处理逻辑：

- 只有 run 分类为 `success` 才允许回写完成。
- 如果 Codex 已经调用 `complete_my_task`，Worker 不重复调用。
- Worker 可通过平台状态确认任务是否已 completed。
- 如果任务未完成但 run 成功，Worker 调用兜底完成接口。

回给主平台：

- 任务状态推进。
- active lease 标记为 completed。
- WorkerRun 标记为 success。
- 写入 `task_completed` 事件。

验收标准：

- 任务完成后平台任务状态为 `completed`。
- 同一任务重复回写不会造成异常状态。

### REQ-WORKER-012：缺陷修复回写

目标：

缺陷修复完成并验证通过后，将缺陷状态回写为待回归。

数据来源：

- Run 结果。
- 缺陷 ID。
- 租约信息。

获取方式：

优先由 Codex 通过 MCP 调用：

```text
agentsprint.fix_bug(bug_id)
```

Worker 兜底可直接调用主平台 API：

```text
POST /mvp/bugs/{id}/fix
```

处理逻辑：

- 只有 run 分类为 `success` 才允许回写已修复。
- 如果 Codex 已经调用 `fix_bug`，Worker 不重复调用。
- 修复成功后 active lease 标记为 completed。

回给主平台：

- 缺陷状态推进到 `fixed_ready_regression`。
- WorkerRun 标记为 success。
- 写入 `bug_fixed` 事件。

验收标准：

- 缺陷状态正确推进。
- 需求健康状态按现有业务规则更新。

### REQ-WORKER-013：失败和阻塞回写

目标：

当 Worker 或 Codex 无法完成任务时，主平台能看到失败原因和证据。

数据来源：

- Run 分类。
- stdout/stderr/final.md。
- 异常堆栈。
- 超时和取消状态。

获取方式：

Worker 直接调用主平台：

```text
POST /worker/runs/{runId}/fail
POST /worker/events
```

处理逻辑：

- 不调用 `complete_my_task` 或 `fix_bug`。
- 根据失败类型决定是否释放租约。
- `blocked` 保留任务租约或标记需要人工介入。
- `codex_failed`、`timeout` 可释放或延长租约，具体由平台策略控制。

回给主平台：

```json
{
  "runId": "run-xxx",
  "status": "blocked",
  "reason": "MCP endpoint unavailable",
  "finalMessage": "无法连接 MCP，不能获取任务详情。",
  "stdoutTail": "...",
  "stderrTail": "..."
}
```

验收标准：

- 失败任务不会被误标记为完成。
- 平台能看到阻塞原因。
- 失败日志能定位到 run。

## 8. 日志与审计需求

### REQ-WORKER-014：本地运行目录落盘

目标：

每次 Codex 执行都有可追溯的本地运行目录。

数据来源：

- WorkerRun。
- CodexProcessRunner。

目录结构：

```text
/runs/{runId}/
  run.json
  prompt.txt
  stdout.log
  stderr.log
  final.md
  events.jsonl
```

处理逻辑：

- run 开始前创建目录。
- 运行中持续追加日志。
- 运行结束写入 `run.json`。
- 敏感字段脱敏。

回给主平台：

- 上传或回写 run 文件路径。
- 回写 stdout/stderr tail。
- 回写 final.md 摘要。

## 10. 异常与恢复需求

### REQ-WORKER-021：任务租约恢复

目标：

Worker 重启后能识别自己是否有未完成租约，避免重复领取或丢任务。

数据来源：

- 主平台 active leases。
- Worker 本地 `/runs`。
- Worker session。

获取方式：

Worker 启动后调用：

```text
GET /worker/sessions/{sessionId}/active-lease
```

或复用现有：

```text
GET /mvp/leases/active
```

处理逻辑：

- 如果存在当前 worker 的 active lease，优先恢复。
- 如果本地有对应 run 且状态 running，标记为 interrupted。
- 平台策略决定继续、释放或人工处理。

回给主平台：

- 上报 `lease_recovery_checked`。
- 如需释放，调用 release lease 接口。

验收标准：

- 容器重启后不会创建重复租约。
- 被中断的 run 在平台可见。

### REQ-WORKER-022：安全脱敏

目标：

日志、事件、prompt 和 run 记录不能泄漏敏感信息。

数据来源：

- 环境变量。
- stdout/stderr。
- config.toml。
- prompt。

处理逻辑：

- 对以下内容脱敏：
  - `OPENAI_API_KEY`
  - `AGENTSPRINT_AGENT_TOKEN`
  - `Authorization`
  - `password`
  - `private key`
  - 数据库连接串
- 保留前后 4-8 位用于排障。

回给主平台：

- 所有事件和 run 摘要使用脱敏后的内容。

验收标准：

- 搜索日志文件，不出现完整 key/token。
- 平台事件中不出现完整 key/token。

## 11. 平台托管配置与日志回推需求

### REQ-WORKER-023：平台托管数字员工环境变量

目标：

数字员工的模型配置、API Key、API 网关、模型名称、MCP 地址等敏感或可变配置统一由主平台维护，减少在服务器 `.env`、compose 文件和外部目录中暴露的风险。

数据来源：

- 主平台数字员工配置页面。
- 主平台 Secret 管理。
- Worker 启动注册请求。

平台需要维护的配置：

| 配置 | 说明 |
| --- | --- |
| `OpenAiApiKeySecretId` | API Key 的 Secret 引用，不在普通列表中明文展示。 |
| `OpenAiBaseUrl` | API 网关地址，例如 `https://devai.yunsee.cn`。 |
| `Model` | Codex 默认模型，例如 `gpt-5.4`。 |
| `SandboxMode` | Codex 沙箱模式，例如 `workspace-write`。 |
| `McpEndpoint` | AgentSprint MCP 地址。 |
| `AgentTokenId` | 数字员工绑定的 Agent Token。 |
| `ProjectScope` | 可处理项目范围。 |
| `MaxRunMinutes` | 单次运行最大时长。 |
| `PollIntervalSeconds` | 默认轮询间隔。 |
| `AllowedTools` | 后续可扩展的工具白名单。 |

获取方式：

Worker 启动后先使用最小 bootstrap token 注册，再调用：

```text
GET /worker/sessions/{sessionId}/runtime-config
```

建议返回：

```json
{
  "workerId": "worker-1",
  "model": "gpt-5.4",
  "openAiBaseUrl": "https://devai.yunsee.cn",
  "openAiApiKey": "只在启动/刷新时短期返回，或返回一次性凭证",
  "mcpEndpoint": "http://mcp:5010/mcp",
  "agentToken": "只在启动/刷新时短期返回，或返回一次性凭证",
  "sandboxMode": "workspace-write",
  "maxRunMinutes": 60,
  "pollIntervalSeconds": 15,
  "configVersion": 3
}
```

处理逻辑：

- 平台保存 Secret 时必须加密存储。
- 管理端列表只展示 Key 前后缀和 Secret 状态，不展示完整明文。
- Worker 拉取配置后，在容器内生成 `/codex-home/config.toml`。
- API Key 和 Agent Token 优先保存在进程环境或临时文件，不写入 compose 文件。
- 如果必须写入 Codex 登录态，由 `codex login --with-api-key` 写入 `/codex-home/auth.json`，不回传明文。
- 配置包含 `configVersion`，Worker 心跳时上报当前版本。
- 平台配置变更后下发 `reload_config` 命令。

回给主平台：

- Worker 上报 `runtime_config_loaded` 事件。
- 心跳携带 `configVersion`、`model`、`openAiBaseUrl` 的脱敏摘要。
- 配置加载失败时上报 `runtime_config_failed`。

安全要求：

- `.env` 只保留 bootstrap token 或短期凭证。
- 不在 `docker-compose.yml` 写 API Key。
- 不在 `config.toml` 写 API Key。
- 不在 WorkerRun、WorkerEvent、stdout、stderr 中写完整 Secret。
- 支持 Secret 轮换，旧配置版本失效后 Worker 重新拉取。

验收标准：

- 管理端可为每个数字员工配置 API 网关、模型和 Secret。
- Worker 启动后能从平台拉取配置并生成 Codex 配置。
- 修改平台模型或网关后，Worker 收到 `reload_config` 并生效。
- 在远端服务器 `.env` 和 compose 文件中不出现完整 API Key。

### REQ-WORKER-024：平台配置下发生成 Codex 配置

目标：

Worker 根据主平台下发的运行配置自动生成 Codex 所需配置，避免人工维护每个容器的 `config.toml`。

数据来源：

- `GET /worker/sessions/{sessionId}/runtime-config`。
- 本地 `/codex-home`。

处理逻辑：

Worker 生成：

```toml
model = "gpt-5.4"
approval_policy = "never"
sandbox_mode = "workspace-write"
openai_base_url = "https://devai.yunsee.cn"

[mcp_servers.agentsprint]
url = "http://mcp:5010/mcp"
bearer_token_env_var = "AGENTSPRINT_AGENT_TOKEN"
required = false
tool_timeout_sec = 120
```

执行步骤：

1. 拉取平台配置。
2. 校验配置版本和必填项。
3. 生成临时 TOML 文件。
4. 进行 TOML 解析校验。
5. 原子替换 `/codex-home/config.toml`。
6. 如 API Key 更新，执行 `codex login --with-api-key`。
7. 上报配置加载结果。

回给主平台：

```json
{
  "eventType": "codex_config_applied",
  "payload": {
    "configVersion": 3,
    "model": "gpt-5.4",
    "openAiBaseUrl": "https://devai.yunsee.cn",
    "mcpEndpoint": "http://mcp:5010/mcp"
  }
}
```

验收标准：

- 删除本地 `config.toml` 后重启 Worker，能从平台恢复配置。
- 平台下发错误 TOML 字段时 Worker 不覆盖旧配置，并上报失败。
- API Key 轮换后 Codex 登录状态刷新成功。

### REQ-WORKER-025：运行日志全量回推

目标：

每个数字员工执行任务时，不只回传状态码，还要把“怎么问、怎么答、怎么做、结果是什么”回推给主平台，形成可审计的任务执行记录。

数据来源：

- `/runs/{runId}/prompt.txt`
- `/runs/{runId}/stdout.log`
- `/runs/{runId}/stderr.log`
- `/runs/{runId}/final.md`
- `/runs/{runId}/events.jsonl`
- Codex session jsonl
- Worker 内部事件

需要回推的内容：

| 内容 | 说明 |
| --- | --- |
| `prompt` | Worker 交给 Codex 的启动问题，即“怎么问”。 |
| `assistantFinal` | Codex 最终回答，即“怎么答”。 |
| `stdoutChunks` | Codex 执行过程输出。 |
| `stderrChunks` | Codex 错误输出。 |
| `toolOrCommandEvents` | Codex 过程中执行的命令、工具或关键动作摘要，即“怎么做”。 |
| `resultSummary` | Worker 解析后的结果摘要。 |
| `artifacts` | final.md、日志文件、截图、测试报告等产物路径或上传 ID。 |

获取方式：

Worker 在运行中分段上传日志：

```text
POST /worker/runs/{runId}/log-chunks
```

建议请求：

```json
{
  "stream": "stdout",
  "sequence": 12,
  "content": "codex output text",
  "offset": 4096,
  "createdAt": "2026-06-11T14:31:40Z"
}
```

运行结束后上传完整摘要：

```text
POST /worker/runs/{runId}/complete-report
```

建议请求：

```json
{
  "prompt": "请调用 agentsprint.get_task_prompt...",
  "assistantFinal": "已完成任务并通过测试。",
  "resultSummary": "修改了用户列表分页逻辑，dotnet test 通过。",
  "exitCode": 0,
  "status": "success",
  "startedAt": "...",
  "completedAt": "...",
  "artifacts": [
    {
      "type": "final",
      "path": "/runs/run-xxx/final.md"
    },
    {
      "type": "stdout",
      "path": "/runs/run-xxx/stdout.log"
    }
  ]
}
```

处理逻辑：

- 日志分段必须带 `sequence`，平台按序重建。
- 单段内容限制大小，例如 16KB 或 64KB。
- 大文件只上传 tail、摘要或对象存储引用。
- prompt 和日志入库前做敏感信息脱敏。
- Codex session jsonl 可解析出 user message、assistant final、token 用量、session id。
- 对命令执行过程，只保留命令摘要、退出码、耗时和关键输出，不默认全量保存可能泄密的环境变量。

回给主平台：

- 平台落库 `WorkerRunLogChunk`。
- 平台更新 `WorkerRun.Prompt`、`WorkerRun.FinalMessage`、`WorkerRun.ResultSummary`。
- 平台记录 `WorkerRunArtifact`。

验收标准：

- 平台可查看一次 run 的完整问答：用户/Worker 问了什么，Codex 答了什么。
- 平台可查看执行过程：关键命令、工具调用、stdout/stderr 摘要。
- 运行失败时平台能看到失败前最后日志。
- 日志中不出现完整 API Key、Agent Token、数据库密码。

## 13. MCP 后期整改方案

### 13.1 整改定位

现有 MCP 服务保持兼容，不直接破坏当前 Codex 开发流。

当前 MCP 的主要定位是：

```text
开发人员 Codex 终端
  -> 连接 AgentSprint MCP
  -> 获取任务上下文
  -> 推进单条开发任务或缺陷
  -> 回写完成状态
```

后期整改新增一组团队协作型 MCP 工具，定位是：

```text
产品经理 AI 终端 / 项目经理 AI 终端 / 架构师 AI 终端 / 测试 AI 终端
  -> 连接 AgentSprint MCP
  -> 提交需求、拉取待评审、发起评审、指派数字员工、查询进展
  -> 主平台统一落库、审计、权限控制
```

设计原则：

- 不修改现有工具的语义。
- 新增工具按角色和场景扩展。
- 所有写入类工具必须做权限校验。
- 所有外部 AI 终端提交的内容必须落库并保留来源。
- 产品经理 AI 可以提交需求，但不能绕过评审和拆解流程。
- AI 终端可以指派数字员工，但最终任务租约仍由主平台控制。

### 13.2 新增 MCP 工具分组

建议按业务域扩展 MCP 工具：

| 分组 | 用途 |
| --- | --- |
| `product_*` | 产品经理 AI 终端提交需求、补充需求、查询需求。 |
| `review_*` | 拉取待评审需求、提交评审意见、推动评审结果。 |
| `worker_*` | 查询数字员工、指派数字员工、查看运行进度。 |
| `team_*` | 查询项目成员、角色、端、模块和当前工作负载。 |
| `delivery_*` | 查询需求推进进度、测试状态、缺陷状态和交付结果。 |

### 13.3 产品经理 AI 需求提交工具

#### REQ-MCP-001：提交产品需求

工具名：

```text
product_submit_requirement
```

使用场景：

产品经理在自己的 AI 终端中梳理需求后，直接通过 MCP 提交到 AgentSprint。

数据怎么来：

- 产品经理 AI 终端输入。
- 附件或上下文文档摘要。
- 项目、端、模块选择。
- 产品经理身份 Token。

参数：

```json
{
  "project_id": "project-1",
  "endpoint_id": "admin",
  "module_id": "system-user",
  "title": "用户列表增加批量禁用功能",
  "description": "产品经理整理后的需求正文",
  "acceptance_criteria": [
    "可以勾选多个用户",
    "点击批量禁用后需要二次确认",
    "禁用后用户不能登录"
  ],
  "priority": 2,
  "source": "pm-ai-terminal",
  "attachments": [
    {
      "name": "prd.md",
      "url": "https://..."
    }
  ]
}
```

数据怎么取：

- MCP 从 Authorization 中解析当前用户。
- 调用主平台项目成员和权限服务。
- 校验当前用户是否是项目产品经理、项目经理或具备需求创建权限。
- 查询项目、端、模块是否存在且可用。

数据怎么处理：

- 创建 `SprintRequirement` 草稿或待评审需求。
- 保存验收标准。
- 保存来源 `source=pm-ai-terminal`。
- 保存原始 AI 提交内容快照。
- 自动补充项目参与人关系。
- 根据配置决定是否自动进入 `pending_review`。

怎么回给主平台：

- 主平台落库需求。
- 写入 `RequirementEvent`：`requirement_submitted_by_ai_terminal`。
- 返回需求 ID、状态、下一步建议。

返回：

```json
{
  "requirement_id": "req-001",
  "status": "pending_review",
  "next_action": "wait_for_review",
  "review_required": true
}
```

验收标准：

- 产品经理 AI 终端能直接创建需求。
- 非项目成员不能创建该项目需求。
- 提交内容可在平台需求详情中追溯来源。

#### REQ-MCP-002：补充或修订产品需求

工具名：

```text
product_update_requirement
```

使用场景：

评审前或评审驳回后，产品经理 AI 终端根据反馈补充需求。

数据怎么来：

- 需求 ID。
- 修订后的标题、描述、验收标准。
- AI 终端补充说明。

参数：

```json
{
  "requirement_id": "req-001",
  "title": "修订后的标题",
  "description": "修订后的描述",
  "acceptance_criteria": [],
  "change_summary": "根据架构师意见补充权限边界"
}
```

数据怎么取：

- 根据 `requirement_id` 查询需求。
- 校验当前用户是否为创建人、产品经理、项目经理或有编辑权限。
- 校验需求状态是否允许编辑。

数据怎么处理：

- 更新需求正文。
- 写入修订历史。
- 如需求处于 rejected，可重新进入待评审。

怎么回给主平台：

- 更新需求记录。
- 写入 `RequirementEvent`：`requirement_updated_by_ai_terminal`。

验收标准：

- 已进入开发中的需求不能被产品经理 AI 直接修改正文。
- 每次修改都有历史记录和来源。

```text
review_list_my_pending_requirements
```

使用场景：

项目经理、架构师或评审人通过 AI 终端拉取待评审需求列表。

数据怎么来：

- 当前用户 Token。
- 可选项目过滤。
- 可选角色过滤。

参数：

```json
{
  "project_id": "project-1",
  "limit": 20,
  "status": "pending_review"
```

数据怎么取：

- MCP 根据当前用户查询其参与项目。
- 查询待该用户评审的需求。
- 关联需求、项目、端、模块、提交人、优先级。

数据怎么处理：

- 按优先级和提交时间排序。
- 返回需求摘要，不返回敏感凭证。

怎么回给主平台：


验收标准：

- 评审人 AI 终端能拉取自己的待评审列表。

#### REQ-MCP-004：获取需求评审上下文

工具名：

```text
review_get_requirement_context
```

使用场景：

评审人 AI 终端在评审前拉取完整需求、历史修改、项目上下文和相关约束。

参数：

```json
{
  "requirement_id": "req-001"
}
```

数据怎么取：

- 查询需求详情。
- 查询验收标准。

- 查询 Skill 或技术栈约束。

数据怎么处理：

验收标准：

- AI 终端能基于返回上下文生成评审建议。
- 返回内容不包含 Secret。

#### REQ-MCP-005：提交需求评审意见

工具名：

```text
review_submit_requirement_decision
```

使用场景：

评审人通过 AI 终端提交通过、驳回或需要补充的意见。

参数：

```json
{
  "requirement_id": "req-001",
  "decision": "approved",
  "comment": "验收标准清晰，可进入拆解",
  "risk_level": "low",
  "need_architect_review": false,
  "suggest_split": true
}
```

数据怎么取：

- 查询需求状态。
- 查询当前用户是否是该需求评审人。
- 查询当前需求评审规则。

数据怎么处理：

- 写入评审记录。
- 如果全员通过，推进需求状态到可拆解。
- 如果驳回，推进需求状态到 rejected。
- 如果需要补充，返回产品经理待补充。

怎么回给主平台：

- 更新评审记录。
- 更新需求状态。
- 写入 `RequirementEvent`。
- 可触发后续 command，例如通知产品经理 AI 终端。

验收标准：

- 评审意见通过 MCP 提交后，平台需求状态同步变化。
- 非评审人不能提交评审决策。

项目经理或产品经理 AI 终端查询当前项目可用的数字员工。

#### REQ-MCP-007：指派数字员工推进任务

工具名：

```text
worker_assign_task
```

使用场景：

产品经理或项目经理 AI 终端将某条已拆解任务交给指定数字员工执行。

参数：

```json
{
  "task_id": "task-001",
  "worker_id": "worker-1",
  "priority": 1,
  "start_mode": "enqueue",
  "instruction": "优先完成后端接口和测试"
}
```

数据怎么取：

- 查询任务详情。
- 查询 Worker 状态。
- 查询当前用户是否具备指派权限。
- 查询任务是否允许被数字员工处理。

数据怎么处理：

- 如果任务未指派，指派给数字员工绑定的系统用户或 Agent 用户。
- 创建 `WorkerCommand`：`start_task` 或 `claim_next_work`。
- 如果 `start_mode=immediate` 且 Worker idle，则下次心跳立即执行。
- 如果 Worker busy，则进入队列。

怎么回给主平台：

- 更新任务负责人或数字员工归属。
- 创建 WorkerCommand。
- 写入 `task_assigned_to_worker` 事件。

返回：

```json
{
  "command_id": "cmd-001",
  "worker_id": "worker-1",
  "task_id": "task-001",
  "status": "queued"
}
```

验收标准：

- 产品经理或项目经理可通过 MCP 指派数字员工。
- 普通开发不能越权指派。
- 被指派任务最终由 Worker 心跳领取并执行。

#### REQ-MCP-008：指派数字员工推进需求拆解

工具名：

```text
worker_assign_requirement_decomposition
```

使用场景：

评审通过后，产品经理或项目经理 AI 终端要求数字员工根据需求拆解开发任务。

参数：

```json
{
  "requirement_id": "req-001",
  "worker_id": "worker-1",
  "task_count": 3,
  "instruction": "按前端、后端、测试验证拆分"
}
```

数据怎么取：

- 查询需求详情和评审状态。
- 查询项目端、模块、技能配置。
- 查询 Worker 可用性。

数据怎么处理：

- 创建 `WorkerCommand`：`decompose_requirement`。
- 或调用已有拆解服务创建任务，再指派给数字员工。
- 拆解结果必须落库为正式任务，不能只停留在 AI 回复中。

怎么回给主平台：

- 创建拆解任务或拆解命令。
- 写入 `requirement_decomposition_requested` 事件。

验收标准：

- 已评审通过的需求可由 AI 终端触发拆解。
- 未评审通过的需求不能被拆解。
- 拆解结果在任务大厅可见。

### 13.8 权限和安全整改

建议权限：

| `delivery_get_requirement_progress` | 是 | 是 | 是 | 是 | 是 | 否 |

### 13.9 与现有开发流的关系

保持现有 Codex 开发流：

```text
开发人员
  -> Codex
  -> AgentSprint MCP
  -> get_task_prompt
  -> complete_my_task
```

新增团队协作流：

```text
产品经理 AI 终端
  -> AgentSprint MCP
  -> product_submit_requirement
  -> review_list_my_pending_requirements
  -> worker_assign_task
  -> delivery_get_requirement_progress
```

两条线并存：

- 开发人员仍可基于 Codex 推进单条需求。
- 产品经理、项目经理、测试可以通过 AI 终端参与团队协作。
- 数字员工由平台统一指派和调度。
- 现有 MCP 工具不改语义，只新增团队协作工具。
