# 数字员工受控端开发排程

## 1. 目标定位

数字员工受控端是运行在独立容器中的 .NET 10 Worker 程序，用于承载一个 Codex 数字员工实例。

核心约束：

- 一个容器只运行一个数字员工。
- 一个数字员工只绑定一个 `CODEX_HOME`、一个工作区根目录、一个运行日志目录和一个 Agent 身份。
- Worker 负责调度 Codex CLI，不替代 AgentSprint API 和 AgentSprint MCP。
- AgentSprint API 负责任务、租约、权限、配置和审计。
- AgentSprint MCP 负责给 Codex 暴露工具和任务上下文。
- Codex CLI 负责实际的 coding agent 执行。

目标闭环：

```text
启动 Worker
  -> 注册数字员工会话
  -> 心跳上报
  -> 领取任务或接收命令
  -> 启动 codex exec
  -> 采集日志和最终响应
  -> 回写任务状态
  -> 继续轮询
```

## 2. 总体架构

建议新增后端项目：

```text
src/api/AgentSprint.Worker/
```

建议组件：

| 组件 | 职责 |
| --- | --- |
| `Program.cs` | Worker 宿主入口、依赖注入、配置加载 |
| `WorkerOptions` | 读取数字员工运行配置 |
| `AgentSprintWorkerService` | 主循环，继承 `BackgroundService` |
| `AgentSprintApiClient` | 调用 AgentSprint API |
| `WorkerSessionManager` | 注册、心跳、下线、会话状态管理 |
| `WorkerCommandLoop` | 轮询平台命令和任务 |
| `CodexProcessRunner` | 启动和监控 `codex exec` |
| `WorkerRunLogger` | 保存 stdout、stderr、final.md、运行摘要 |
| `WorkspaceManager` | 准备项目工作区、仓库和分支 |

基础运行目录：

```text
/codex-home
/workspaces
/runs
```

基础配置：

```json
{
  "Worker": {
    "WorkerId": "worker-1",
    "WorkerName": "math-codex-worker-1",
    "ProjectId": "",
    "ProjectCode": "",
    "WorkspaceRoot": "/workspaces",
    "RunsRoot": "/runs",
    "CodexHome": "/codex-home",
    "PollIntervalSeconds": 15,
    "IdleMaxIntervalSeconds": 180,

  },
  "AgentSprint": {
    "ApiBaseUrl": "http://api:5000",
    "AgentToken": ""
  }
}
```

## 3. 通信模型

第一阶段采用可靠性优先的通信方式：

```text
HTTP 注册 + HTTP 心跳 + command polling + event 上报
```

后续增强：

```text
SignalR/WebSocket 只做唤醒通知
Actor/Supervisor 只做服务端内存态组织
数据库中的 Session、Command、Run、Lease、Event 仍是事实源
```

不建议第一版只依赖长连接。原因：

- 容器重启会丢连接。
- 平台重启会丢内存态。
- 长任务执行期间网络断开不等于任务失败。
- 多副本平台部署时连接路由复杂。

## 4. 执行主循环

Worker 启动：

```text
校验 codex/git/dotnet/node 可用
校验 CODEX_HOME/config.toml 存在
校验 codex login status
注册 worker session
发送 online 心跳
```

Worker 循环：

```text
发送 heartbeat
读取 commands 或 claim_next_work
无任务则按 polling interval 等待
有任务则创建 WorkerRun
准备 workspace
启动 codex exec
持续采集 stdout/stderr
保存 final.md
分析退出码和最终结果
回写 complete_my_task / fix_bug / append_agent_event
继续下一轮
```

Codex MVP 命令：

```bash
codex exec \
  --cd /workspaces/<project-code> \
  --sandbox workspace-write \
  --skip-git-repo-check \
  --output-last-message /runs/<run-id>/final.md \
  "请调用 agentsprint.get_task_prompt 获取 task_id=<task-id> 后完成任务。"
```

有真实仓库后再移除 `--skip-git-repo-check`。

## 5. 阶段排程

### 第 1 阶段：最小可运行 Worker

预计周期：1-2 天。

开发内容：

- 新增 `AgentSprint.Worker` 项目。
- 加入 Worker 配置模型。
- 实现 `BackgroundService` 主循环骨架。
- 启动时检查 `codex --version`、`git --version`、`dotnet --version`、`node --version`。
- 检查 `codex login status`。
- 实现 `CodexProcessRunner`。
- 支持手动传入 prompt 执行 `codex exec`。
- 保存运行产物：
  - `/runs/{runId}/stdout.log`
  - `/runs/{runId}/stderr.log`
  - `/runs/{runId}/final.md`
  - `/runs/{runId}/run.json`

验收标准：

- `docker compose up -d worker` 可以启动 Worker。
- Worker 可以执行测试 prompt `你好`。
- `/runs/{runId}/final.md` 出现模型响应。
- stdout/stderr 能落盘。

### 第 2 阶段：接入 AgentSprint API

预计周期：2-3 天。

开发内容：

- 实现 `AgentSprintApiClient`。
- 新增或复用 Worker 注册接口。
- 新增平台模型：
  - `DigitalWorker`
  - `WorkerSession`
  - `WorkerHeartbeat`
  - `WorkerRun`
  - `WorkerEvent`
- Worker 启动时注册 session。
- Worker 定时上报 heartbeat。
- Worker 执行 run 时上报事件。
- 平台支持查看 worker online、idle、busy、offline。

验收标准：

- 平台能看到数字员工在线。
- 心跳时间持续刷新。
- 执行一次测试 run 后，平台能查询到 run 记录。
- Worker 停止后状态能进入 offline 或 expired。

### 第 3 阶段：任务领取闭环

预计周期：2-3 天。

开发内容：

- Worker 调用 `claim_next_work`。
- 支持任务类型：
  - development task
  - bug fix
- 根据任务生成 Codex prompt。
- Codex 通过 MCP 调用 `get_task_prompt` 拉取任务详情。
- Codex 完成后由 Worker 或 Codex 调用：
  - `complete_my_task`
  - `fix_bug`
  - `append_agent_event`
- 失败时写入失败事件和失败摘要。

验收标准：

- 平台任务能从 `assigned` 或 `in_progress` 推进到 `completed`。
- Bug 能从 `open` 或 `fixing` 推进到待回归状态。
- 失败任务能看到失败原因、stdout、stderr 和 final.md。
- Worker 空闲后继续按 polling 策略轮询。

### 第 4 阶段：可靠性和恢复

预计周期：3-5 天。

开发内容：

- 支持 run 超时控制。
- 支持取消当前 Codex 进程。
- 支持 `stop_after_current`。
- 支持 `offline_requested`。
- 支持租约续期。
- 支持 Worker 崩溃后恢复状态。
- 禁止同一 Worker 并发执行多个 run。
- stdout/stderr 分段上报。
- run 结果分类：
  - success
  - codex_failed
  - test_failed
  - timeout
  - cancelled
  - blocked
  - lease_lost

验收标准：

- 杀掉容器后重启，不会重复领取同一任务。
- 长任务超时后能被标记为失败或阻塞。
- 平台可以请求 Worker 停止当前任务。
- Worker 下线后平台租约可以过期或释放。

### 第 5 阶段：多数字员工部署

预计周期：3-5 天。

开发内容：

- Compose 支持多个 Worker service。
- 每个 Worker 独立：
  - `.env`
  - `/codex-home`
  - `/workspaces`
  - `/runs`
  - Agent Token
- 平台按项目、端、模块、技能和优先级分配任务。
- 支持 worker/project/skill 维度的任务范围控制。
- 支持并发数量和资源限制配置。

验收标准：

- `worker-1`、`worker-2` 可以同时在线。
- 两个 Worker 不会领取同一个任务。
- 每个 Worker 只访问自己的工作区和 Token。
- 停止某个 Worker 不影响其他 Worker。

## 6. 后续增强

### SignalR/WebSocket 唤醒

用途：

- 平台有新任务时立即通知 Worker。
- 平台下发停止、离线、刷新配置等命令。

约束：

- WebSocket 只做通知。
- Worker 收到通知后仍通过 HTTP 拉取命令详情。
- 命令执行结果仍通过 HTTP ACK 和事件上报落库。

### Actor/Supervisor

用途：

- 服务端内部管理 Worker 运行视图。
- 快速聚合在线状态、当前任务和命令队列。

约束：

- Actor 不是事实源。
- 数据库里的 Session、Run、Event、Lease 才是事实源。

## 7. 安全要求

- API Key、Agent Token 不写入 Git。
- API Key、Agent Token 不写入 prompt。
- API Key、Agent Token 通过 `.env` 或 Secret 管理注入。
- 每个数字员工使用独立 Agent Token。
- 每个数字员工使用独立 `CODEX_HOME`。
- 默认使用 `sandbox_mode = workspace-write`。
- `danger-full-access` 只能在强隔离测试容器中使用。
- 不共享 `/workspaces`。
- 不共享 `/codex-home`。

## 8. 当前建议优先级

优先做：

1. `AgentSprint.Worker` 项目骨架。
2. `CodexProcessRunner`。
3. `/runs/{runId}` 日志落盘。
4. Heartbeat + Run 记录落库。
5. `claim_next_work` 到 `complete_my_task` 的最小闭环。

暂缓做：

1. WebSocket 长连接。
2. Actor/Supervisor。
3. 多 Worker 复杂调度。
4. 自动升级镜像。
5. 完整审计报表。
