# 烛照项目自动化协同平台
 简称烛照协同平台,面向 AI Agent / Codex 的敏捷需求接取、开发、测试发布平台。/Agile workflow platform for Codex-style coding agents. 
<img width="2560" height="1389" alt="image" src="https://github.com/user-attachments/assets/f7ff231a-f756-47bc-a924-57736de19321" />



## 自动化流程

 通过模板化提示词结合MCP工具,将原本的开发模式提升到完全自动化的方式
 
 ## 特点
  项目以传统项目施工角度出发,减少了一些在中小型团队中应用的复杂流转流程,与OpenSepc与OpenClaw的区别是该项目是基于团队化,项目化的角度来推动项目研发,整体流程可管控,可统计


 ## 流程

   ### 个人准备
   1. 一个Codex环境可以访问到的git仓库
   2. 手动将项目拉取到你的文件夹中
   3. 在Codex配置项目工作区
   4. 配置完全访问权限(可以不配置,但是遇到问题Codex将会需要你确认,无法全自动化)

   ### 项目准备
   1. 在烛照协同平台建立系统项目
   2. 根据项目信息建立对应的端,例如: 管理端,移动端,IOS等
   3. 为对应的端建立项目需求,并配置是否需要评审
   4. 需求评审(如果需求需要评审则联系评审人评审)
   5. 需求任务拆解(将需求拆解成一条条待办任务)
   6. 任务大厅任务分配(在任务拆解时可以指定人完成,也可以由系统自动指定)

   ### 开始工作
   1. 登录烛照协同平台中的我的任务列表找到优先级最高的任务
   <img width="2560" height="1389" alt="image" src="https://github.com/user-attachments/assets/ca441c79-1925-4827-b33b-004e6624cfbb" />
   
   2. 选中任务并点击任务推进
   
   3. 根据弹出的抽屉复制MCP服务器信息,粘贴给Codex,让Codex配置MCP服务器(仅仅在初次配置/烛照协同平台迁移到其他位置时执行)
   <img width="2560" height="1389" alt="image" src="https://github.com/user-attachments/assets/e44b45e5-4f0b-4a45-82c0-f2a392fd1279" />
   
   4. 复制任务推进提示词,键入Codex项目工作空间聊天框中即可
   <img width="2560" height="1541" alt="image" src="https://github.com/user-attachments/assets/2d4464f6-03c7-4053-bafb-0b11d11031aa" />

   5. Codex可能会将其设为一个目标,并持续进行自动化推进,只要平台内有当前人需要推进的任务即可自动推进,无需人工干预

## 后续规划
   1. 优化现有的功能流程,修复页面操作Bug
   2. 增加基于AI的自动任务拆解并生成任务拆解建议供产品经理拆解任务时选择
   3. 尝试增加数字员工,探索在服务器上安装Codex并使其默认开启自动化,减少对于开发设备的依赖,让产品经理与设计随时随地可以将想法变为现实
   4. 为MCP服务器提供基于需求的Skill加载功能(当前Skill配置已有,MCP服务未调整)
   5. 增加代码审查管理(配置项目各端的代码审查Skill,让AI根据需求与Skill审查研发成果)
   6. 项目配置技术栈细化为技术栈管理功能,为每个技术栈提供模板项目配置,后续新建项目时,引导AI自动建立项目仓库,并配置相关环境
   7. 项目配置增加项目文档管理,项目统计,项目推进视图
   8. 运行环境管理优化,增加sshkey管理,部署脚本在线维护
   9. 增加流水线集成(允许AI产物推动到harbor,Jenkins,远程服务器指定位置,对象存储服务等三方存储系统)
   10. 增加项目运行监控,允许以整体角度观察项目管理中的进度统计,生产环境,测试环境,各类关联组件/服务器的状态监控
   11. 增加AI管理, 将安全管理的令牌管理移动到此,并增加 当前活动自动化会话的管理(MCP已支持), 三方OpenAI协议的对接与配置(类中转站的消息中转与转发)
   12. 增加安全审查管理,对每次AI的对话内容进行安全审计,严防泄密,数据泄露等各类安全合规性要求(前提是10已经完成)

## 注意事项

   1.不建议在已有的项目中推动此平台,这可能会导致代码发生大规模重构/数据库重建等危险性操作, 在尝试此平台时,优先建议尝试新的项目或者微服务架构中的某个单独的服务
   
   2.实际大规模应用请等待正式的V1.0版本,我们将会内置一些Skill以约束规范AI的行为来降低Token的消耗量,目前对于Token的消耗量抵得上普通开发的5-10倍,本人亲身体验1人,3个Codex会话在2天内消耗完3个GPT 20X账户的1周配额
